## Executive Summary

EverionMind is a knowledge-management platform built around **entries** (captured data), **brains** (namespaced knowledge spaces), and **concept graphs** (AI-extracted semantic relationships). The current architecture is in the "works but fragile" zone â€” feature velocity was correctly prioritised, but the accumulated coupling is now creating meaningful risk as the system grows.

This audit identifies six architectural friction points with implementation-ready refactor specs for each. Every finding is grounded in actual code with specific file paths and line numbers.

## Friction Point 1: Everion.tsx â€” 1,171-Line God Component

### What it is

`src/Everion.tsx` is the application root. It is responsible for everything:

- ~45 independent `useState` calls (core data, UI, modals, enrichment, online state, vault, theme)
- All AI enrichment orchestration (`runBulkEnrich`, `handleCreated`, `handleUpdate` callbacks)
- Loading entries and links from the API on brain change (lines 323â€“345)
- Deriving concept graph, filtered entries, sorted timelines (lines 315â€“365)
- Providing `EntriesContext` and `BrainContext` to the tree (lines ~1120)
- Navigation between views (Feed, Ask, Refine, Detail, etc.)

### Root causes

**State colocation without domain separation.** All state lives at the root because early development iterated fast without establishing domain boundaries. UI concerns (modal open/closed, view name, search input) share the same component as data concerns (entries, brain, enrichment queue), which share the same component as orchestration concerns (how enrichment pipelines are triggered).

**Hooks return values re-assembled at root.** `useChat`, `useOfflineSync`, `useBrainHook`, `useEntryActions`, `useNudge`, `useBackgroundCapture`, `useRole`, `useTheme` all return values that Everion.tsx destructures and re-passes downward. The hooks do real work, but Everion.tsx is the glue that wires them together and re-exposes everything.

**The concept graph is derived on every entry mutation** (lines 337â€“340):

```typescript
const conceptGraph: ConceptGraph | null = useMemo(() => {
  const g = loadGraph(activeBrain.id); // localStorage read
  return g.concepts.length > 0 ? g : null;
}, [activeBrain?.id, entries]); // re-runs on every entries change
```

Entries change hundreds of times per session (on load, sync, enrichment). Every change re-reads localStorage and reconstructs the concept map.

### Coupling map

```
Everion.tsx

â”śâ”€ Passes 50+ props to children
â”śâ”€ Contains: entry loading, cache writing, search indexing,

â”‚            before-unload flush, debounced search, online tracking
â””â”€ Provides: EntriesContext, BrainContext
```

### Implementation plan

**Phase A â€” Extract UI state into a `useAppShell` hook**

Move all view/modal/UI state out of Everion.tsx:

```typescript

  const [showCapture, setShowCapture] = useState(false);

  const [showBrainTip, setShowBrainTip] = useState<Brain | null>(null);
  const [selectMode, setSelectMode] = useState(false);
  const [selectedIds, setSelectedIds] = useState<Set<string>>(new Set());
  const [gridViewMode, setGridViewMode] = useState<"grid" | "list">(...);
  const [searchInput, setSearchInput] = useState("");
  const [search, setSearch] = useState("");
  return { view, setView, showCapture, ... };

```

Everion.tsx becomes:

```typescript

const enrichment = useEnrichmentOrchestrator(data, activeBrain);
```

**Phase B â€” Extract data loading into a `useDataLayer` hook**

**Phase C â€” Lift concept graph into a `ConceptGraphProvider`**

- Concept graph re-derives only when `activeBrain.id` changes, not on every entry mutation

## Friction Point 2: Empty Context Abstraction â€” Middleware Theater

### What it is

Both context files are empty wrappers:

```typescript
// src/context/EntriesContext.tsx (6 lines)
export const EntriesContext = createContext<any>(null);
export const useEntries = () => useContext(EntriesContext);

// src/context/BrainContext.tsx (6 lines)
export const BrainContext = createContext<any>(null);
export const useBrain = () => useContext(BrainContext);
```

The actual values are defined inline in `Everion.tsx` (~lines 350â€“370):

```typescript
const entriesValue = useMemo(() => ({
  entries, setEntries, entriesLoaded, selected, setSelected, handleDelete, handleUpdate,
}), [...]);

const brainValue = useMemo(() => ({
  activeBrain, brains, setActiveBrain, refresh, canInvite, canManageMembers, deleteBrain,
}), [...]);
```

### Root causes

The contexts were scaffolded as placeholders but the `<any>` type was never replaced. Because the shape is defined only in Everion.tsx, consumers (`DetailModal`, `useBrain.ts`, etc.) must know the context shape by convention â€” there is no IDE autocomplete, no compile-time safety, and no single documentation source for "what is available in context".

### Coupling consequences

- Any consumer that reads `useEntries()` but expects a field that was renamed or removed fails silently at runtime
- New developers must grep Everion.tsx to learn the context shape
- Mock contexts in tests require manually reconstructing the same undocumented shape
- `setEntries`, `handleUpdate`, and `handleDelete` being in context means any component can mutate root entry state without going through a defined interface

### Implementation plan

**Step 1 â€” Define explicit context interfaces**

```typescript
// src/context/EntriesContext.tsx
export interface EntriesContextValue {
  entries: Entry[];
  entriesLoaded: boolean;
  selected: Entry | null;
  setSelected: (entry: Entry | null) => void;
  handleDelete: (id: string) => void;
  handleUpdate: (id: string, changes: Partial<Entry>, opts?: { silent?: boolean }) => Promise<void>;
}

export const EntriesContext = createContext<EntriesContextValue | null>(null);

export function useEntries(): EntriesContextValue {
  const ctx = useContext(EntriesContext);
  if (!ctx) throw new Error("useEntries must be used within EntriesContext.Provider");
  return ctx;
}
```

```typescript
// src/context/BrainContext.tsx
export interface BrainContextValue {
  activeBrain: Brain | null;
  brains: Brain[];
  setActiveBrain: (brain: Brain) => void;
  refresh: () => Promise<void>;
  canInvite: boolean;
  canManageMembers: boolean;
  deleteBrain: (id: string) => Promise<void>;
}

export const BrainContext = createContext<BrainContextValue | null>(null);

export function useBrainContext(): BrainContextValue {
  const ctx = useContext(BrainContext);
  if (!ctx) throw new Error("useBrainContext must be used within BrainContext.Provider");
  return ctx;
}
```

**Step 2 â€” Remove setEntries from context**

`setEntries` being in context is a write-backdoor. Replace it with the two defined operations: `handleDelete` and `handleUpdate`. Consumers that currently call `setEntries` directly should go through those methods. This is the boundary: the context exposes _what you can do_, not _how state is stored_.

**Step 3 â€” Update Everion.tsx provider values to match the interfaces**

The `useMemo` shapes in Everion.tsx get typed:

```typescript
const entriesValue: EntriesContextValue = useMemo(
  () => ({
    entries,
    entriesLoaded,
    selected,
    setSelected,
    handleDelete,
    handleUpdate,
  }),
  [entries, entriesLoaded, selected, setSelected, handleDelete, handleUpdate],
);
```

TypeScript will now fail at compile time if the shape drifts.

**Affected files:**

- `src/context/EntriesContext.tsx` â€” add explicit interface, remove `any`
- `src/context/BrainContext.tsx` â€” add explicit interface, remove `any`
- `src/Everion.tsx` â€” type the `useMemo` context values
- Any consumer using `useEntries().setEntries` â€” migrate to `handleUpdate`

**Verification criteria:**

- `any` is absent from both context files
- `tsc --noEmit` passes with no `any` suppressions in context consumers
- A test for `useEntries()` outside a provider throws with a readable error message

## Friction Point 3: AI Enrichment â€” Scattered Orchestrators with Race Conditions

### What it is

The enrichment pipeline is split across four independent orchestrators with overlapping responsibilities and no coordinated sequencing:

| Module                           | Responsibility                                     | Lines |
| -------------------------------- | -------------------------------------------------- | ----- |
| `src/lib/enrichEntry.ts`         | 4-phase: parse â†’ embed â†’ concepts â†’ insight        | 159   |
| `src/lib/brainConnections.ts`    | Per-entry concept extraction, batch graph building | 349   |
| `src/hooks/useRefineAnalysis.ts` | Audit suggestions + graph update in one AI call    | 1,135 |
| `src/hooks/useChat.ts`           | Feeds query terms back into concept graph          | 368   |

### The race condition

`brainConnections.ts` uses a module-level lock to serialize graph writes:

```typescript
// src/lib/brainConnections.ts line 14
let _graphLock: Promise<void> = Promise.resolve();

export async function extractEntryConnections(entry: EntryRef, brainId: string) {
  _graphLock = _graphLock.then(() => _doExtractEntryConnections(entry, brainId));
  return _graphLock;
}
```

The lock is module-level, so it serializes ALL calls across ALL brains globally. But more critically, three other places can write to the graph without going through this lock:

1. `useRefineAnalysis.ts` lines ~800â€“820 â€” writes graph after AI audit call without `_graphLock`
2. `useChat.ts` â€” calls `feedQueryToGraph()` directly
3. `enrichEntry.ts` â€” calls `extractEntryConnections()` (goes through lock) but also calls `generateEntryInsight()` which may write enrichment metadata in parallel

**Concrete failure scenario:**

1. User captures entry A â€” `enrichEntry` runs, calls `extractEntryConnections` â†’ acquires lock, loads graph from DB
2. While lock is held, user opens Refine â€” `useRefineAnalysis.analyze()` runs concurrently, calls `loadGraphFromDB` â†’ gets stale pre-lock graph
3. Both save: whichever saves last silently overwrites the other's concepts

### onUpdate callback coupling

`enrichEntry.ts` receives mutations as a callback:

```typescript
export async function enrichEntry(
  entry: Entry,
  brainId: string,
  onUpdate: (id: string, changes: any) => Promise<void>, // untyped
): Promise<void>;
```

The callback is bound in Everion.tsx to `handleUpdate(id, changes, {silent: true})`. `enrichEntry` and Everion.tsx are tightly coupled through this untyped contract. If the callback shape changes, the compiler won't catch it.

### Duplicated graph writes

`useRefineAnalysis.ts` (lines ~800â€“820) contains its own graph extraction and save that duplicates `brainConnections.ts`:

```typescript
if (activeBrain?.id && (p.concepts || p.relationships)) {
  const newConcepts = p.concepts ? extractConcepts(p.concepts) : [];
  const newRels = p.relationships ? extractRelationships(p.relationships) : [];
  const existing = await loadGraphFromDB(activeBrain.id);
  const merged = mergeGraph(existing, { concepts: newConcepts, relationships: newRels });
  await saveGraphToDB(activeBrain.id, merged); // no lock, races with brainConnections
}
```

### Implementation plan

**Extract a typed `EnrichmentUpdate` interface**

```typescript
// src/lib/enrichEntry.ts â€” replace `any` callback
export interface EnrichmentUpdate {
  type?: EntryType;
  content?: string;
  metadata?: EntryMetadata;
}

export async function enrichEntry(
  entry: Entry,
  brainId: string,
  onUpdate: (id: string, changes: EnrichmentUpdate) => Promise<void>,
): Promise<void>;
```

**Centralise graph writes through a single `GraphWriter`**

Create `src/lib/graphWriter.ts` as the single place that holds the per-brain lock and executes all graph mutations:

```typescript
// src/lib/graphWriter.ts
const locks = new Map<string, Promise<void>>();

async function withBrainLock<T>(brainId: string, fn: () => Promise<T>): Promise<T> {
  const existing = locks.get(brainId) ?? Promise.resolve();
  let resolve!: () => void;
  const next = new Promise<void>((res) => {
    resolve = res;
  });
  locks.set(brainId, next);
  await existing;
  try {
    return await fn();
  } finally {
    resolve();
    if (locks.get(brainId) === next) locks.delete(brainId);
  }
}

export async function writeConceptsToGraph(
  brainId: string,
  incoming: { concepts: Concept[]; relationships: Relationship[] },
): Promise<void> {
  await withBrainLock(brainId, async () => {
    const existing = await loadGraphFromDB(brainId);
    const merged = mergeGraph(existing, incoming);
    await saveGraphToDB(brainId, merged);
  });
}
```

All three callers (`brainConnections.ts`, `useRefineAnalysis.ts`, future callers) replace their loadâ†’mergeâ†’save pattern with a single `writeConceptsToGraph(brainId, { concepts, relationships })` call. The lock is:

- Per-brain (not global)
- Cleaned up when no longer needed
- The only place that touches `loadGraphFromDB` + `saveGraphToDB` in sequence

**Remove `_graphLock` from `brainConnections.ts`**

After `graphWriter.ts` is in place, delete the module-level lock from `brainConnections.ts` and have all public functions delegate to `writeConceptsToGraph`.

**Affected files:**

- `src/lib/graphWriter.ts` â€” new
- `src/lib/brainConnections.ts` â€” remove `_graphLock`, delegate graph writes to `graphWriter`
- `src/lib/enrichEntry.ts` â€” type the `onUpdate` callback
- `src/hooks/useRefineAnalysis.ts` â€” replace inline loadâ†’mergeâ†’save with `writeConceptsToGraph`
- `src/hooks/useChat.ts` â€” `feedQueryToGraph` should go through `graphWriter` if it writes

**Verification criteria:**

- `_graphLock` does not appear anywhere in the codebase after refactor
- A test simulates two concurrent `writeConceptsToGraph` calls for the same brainId and verifies the second waits for the first
- `EnrichmentUpdate` type is used at all three call sites
- No `any` in enrichEntry callback signature

## Friction Point 4: authFetch â€” Side-Effecting Transport Layer

### What it is

`src/lib/authFetch.ts` (42 lines) is the single HTTP client used by the entire app â€” every `authFetch` call does three things:

```typescript
export async function authFetch(url: string, options: RequestInit = {}): Promise<Response> {
  // 1. Hits Supabase auth.getSession() on every call
  const {
    data: { session },
  } = await supabase.auth.getSession();

  // 2. Makes the actual fetch with Authorization header
  const response = await fetch(url, {
    ...options,
    headers: { ...options.headers, Authorization: `Bearer ${session?.access_token}` },
  });

  // 3. Parses X-Embedding-Usage header as a side effect (usage tracking)
  const embeddingHeader = response.headers.get("X-Embedding-Usage");
  if (embeddingHeader) {
    import("./usageTracker")
      .then((m) => {
        m.recordUsage({ date, type: "embedding", provider, model, embeddingCount: count });
      })
      .catch(() => {}); // fire-and-forget
  }

  return response;
}
```

### Problems

**`getSession()` on every call.** Supabase's `getSession()` reads from localStorage/memory but still involves an async call. With 100+ `authFetch` calls per session (entry loads, AI calls, embed calls, graph saves, link saves), this adds up and burns through Supabase's rate limits on the session endpoint.

**Embedding usage tracking is a hidden side effect.** A function named `authFetch` secretly records AI usage metrics. This violates the single-responsibility principle and makes it impossible to call `authFetch` in a test without either mocking `usageTracker` or accepting silent import failures.

**Embedding header logic is duplicated.** `getEmbedHeaders()` â€” which constructs the headers that activate the server-side embedding path â€” is called separately in at least five places: `useEntryActions.ts`, `enrichEntry.ts`, `useOfflineSync.ts`, `useChat.ts`, and `Everion.tsx`. Each call site has to decide independently whether embedding is available.

### Implementation plan

**Step 1 â€” Cache the session token**

```typescript
// src/lib/authFetch.ts
let _sessionCache: { token: string; expiresAt: number } | null = null;
const SESSION_CACHE_TTL = 4 * 60 * 1000; // 4 minutes (Supabase tokens are 60min)

async function getSessionToken(): Promise<string | null> {
  if (_sessionCache && Date.now() < _sessionCache.expiresAt) {
    return _sessionCache.token;
  }
  const {
    data: { session },
  } = await supabase.auth.getSession();
  if (session?.access_token) {
    _sessionCache = { token: session.access_token, expiresAt: Date.now() + SESSION_CACHE_TTL };
  }
  return session?.access_token ?? null;
}

// Subscribe to session changes to invalidate cache
supabase.auth.onAuthStateChange(() => {
  _sessionCache = null;
});
```

This reduces `getSession()` calls from ~100/session to ~15/session.

**Step 2 â€” Separate usage tracking into a response interceptor**

Remove the `X-Embedding-Usage` side effect from `authFetch`. Instead, create an explicit `trackEmbeddingUsage` function that callers opt into:

```typescript
// src/lib/usageTracker.ts â€” add explicit tracker
export function trackEmbeddingIfPresent(response: Response): void {
  const header = response.headers.get("X-Embedding-Usage");
  if (!header) return;
  // parse and record
}
```

Callers that care about embedding tracking (currently `authFetch`) opt in:

```typescript
// In authFetch â€” just transport, no side effects
export async function authFetch(url: string, options: RequestInit = {}): Promise<Response> {
  const token = await getSessionToken();
  return fetch(url, {
    ...options,
    headers: { ...options.headers, ...(token ? { Authorization: `Bearer ${token}` } : {}) },
  });
}

// Call site in entry creation / AI calls that want tracking:
const response = await authFetch("/api/embed", opts);
trackEmbeddingIfPresent(response);
```

**Step 3 â€” Centralise embed header decision**

Replace the 5 scattered `getEmbedHeaders()` calls with a single utility:

```typescript
// src/lib/embeddingContext.ts
export function shouldUseServerEmbedding(): boolean {
  return !!getEmbedHeaders();
}

export function getEmbeddingHeaders(): Record<string, string> {
  return getEmbedHeaders() ?? {};
}
```

The two call sites in `useChat.ts` that branch on whether embedding headers exist become:

```typescript
if (shouldUseServerEmbedding() && activeBrain?.id) {
  // server path
} else {
  // client fallback path
}
```

**Affected files:**

- `src/lib/authFetch.ts` â€” add session caching, remove usage tracking side effect
- `src/lib/usageTracker.ts` â€” add `trackEmbeddingIfPresent` export
- `src/lib/embeddingContext.ts` â€” new, centralises embedding header decision
- `useChat.ts`, `enrichEntry.ts`, `useOfflineSync.ts`, `useEntryActions.ts` â€” update to use `shouldUseServerEmbedding()` and explicit `trackEmbeddingIfPresent`

**Verification criteria:**

- `authFetch` does not import `usageTracker`
- `getSession()` is called at most once per 4-minute window in a simulated 100-call test
- A test can call `authFetch` with a mocked `fetch` without any `usageTracker` import side effects

## Friction Point 5: ConceptGraph â€” Dual State with No Conflict Resolution

### What it is

The concept graph lives in two places simultaneously:

- **localStorage**: `concept_graph_${brainId}` â€” synchronous read, used for instant UI rendering
- **Supabase** (`user_graphs` table): loaded via `/api/graph`, used as authoritative store

The read/write pattern in `src/lib/conceptGraph.ts`:

```typescript
export async function loadGraphFromDB(brainId: string): Promise<ConceptGraph> {
  const res = await authFetch(`/api/graph?brain_id=...`);
  const graph = await res.json();
  localStorage.setItem(GRAPH_KEY(brainId), JSON.stringify(graph)); // update cache
  return graph;
}

export async function saveGraphToDB(brainId: string, graph: ConceptGraph): Promise<void> {
  localStorage.setItem(GRAPH_KEY(brainId), JSON.stringify(graph)); // write cache
  await authFetch("/api/graph", { method: "POST", body: JSON.stringify({ brain_id, graph }) });
  // If DB write fails: console.error only â€” cache is now ahead of DB
}
```

### The normalization problem

Concept deduplication uses `normalize()`:

```typescript
function normalize(label: string): string {
  return label
    .toLowerCase()
    .trim()
    .replace(/[^a-z0-9\s]/g, "");
}
```

- "Dr. Smith's Practice" â†’ `"dr smiths practice"`
- "Dr Smiths Practice" â†’ `"dr smiths practice"` âś“ (deduplicates correctly)
- "Dr. Smith's practice" â†’ `"dr smiths practice"` âś“
- But: "Smith's Practice" â†’ `"smiths practice"` (different key â€” does NOT deduplicate)

The concept `id` is the normalized key, and the `label` stored is the first-seen spelling. This means:

- Users see "Dr. Smith's Practice" in the UI
- Internally the node is `"dr smiths practice"`
- If AI later produces "Smith Practice" (without the possessive), it creates `"smith practice"` â€” a separate node
- The graph grows duplicates silently, degrading the concept network

### Missing version field

There is no schema version field on `ConceptGraph`. When the data shape changes (new fields, format changes), there is no mechanism to detect that stored data is stale. Migration must be handled by reading stale data and hoping the `mergeGraph` function tolerates it.

### Implementation plan

**Step 1 â€” Add schema version to ConceptGraph**

```typescript
// src/lib/conceptGraph.ts
export interface ConceptGraph {
  version: 2; // bump when schema changes
  concepts: Concept[];
  relationships: Relationship[];
}

const CURRENT_VERSION = 2;

function migrateGraph(raw: unknown): ConceptGraph {
  const g = raw as any;
  if (!g || typeof g !== "object")
    return { version: CURRENT_VERSION, concepts: [], relationships: [] };
  if (!g.version) {
    // v1 had no version field â€” migrate
    return {
      version: CURRENT_VERSION,
      concepts: g.concepts ?? [],
      relationships: g.relationships ?? [],
    };
  }
  return g as ConceptGraph;
}
```

**Step 2 â€” Improve normalization to reduce false non-deduplication**

The current `normalize` strips possessives and punctuation but doesn't handle word-order variation or synonym matching. A more robust approach without adding heavy dependencies:

```typescript
function normalize(label: string): string {
  return label
    .toLowerCase()
    .trim()
    .replace(/[''\u2019s]+\b/g, "") // remove possessives: "Smith's" â†’ "Smith"
    .replace(/[^a-z0-9\s]/g, "") // strip remaining punctuation
    .replace(/\s+/g, " ") // normalise whitespace
    .trim();
}
```

This ensures "Smith's Practice" and "Smiths Practice" both normalize to `"smiths practice"`, matching the "Dr. Smith's Practice" node.

**Step 3 â€” Track in-memory dirty state, don't allow cache-ahead-of-DB silently**

Add a `dirty` flag to detect when the localStorage cache has been written but the DB write failed:

```typescript
const DIRTY_KEY = (brainId: string) => `concept_graph_dirty_${brainId}`;

export async function saveGraphToDB(brainId: string, graph: ConceptGraph): Promise<void> {
  localStorage.setItem(GRAPH_KEY(brainId), JSON.stringify(graph));
  localStorage.setItem(DIRTY_KEY(brainId), "1"); // mark dirty before DB write
  try {
    await authFetch("/api/graph", {
      method: "POST",
      body: JSON.stringify({ brain_id: brainId, graph }),
    });
    localStorage.removeItem(DIRTY_KEY(brainId)); // clear dirty on success
  } catch (err) {
    console.error("[conceptGraph] DB save failed, graph is dirty:", err);
    // dirty flag stays â€” next loadGraphFromDB call will see it and retry
  }
}

export async function loadGraphFromDB(brainId: string): Promise<ConceptGraph> {
  // If dirty, try to flush before loading (optional retry pattern)
  const dirty = localStorage.getItem(DIRTY_KEY(brainId));
  if (dirty) {
    const cached = loadGraph(brainId);
    try {
      await authFetch("/api/graph", {
        method: "POST",
        body: JSON.stringify({ brain_id: brainId, graph: cached }),
      });
      localStorage.removeItem(DIRTY_KEY(brainId));
    } catch {
      /* will retry next time */
    }
  }
  // ... rest of current logic
}
```

**Step 4 â€” Expose graph validation**

```typescript
export function validateGraph(graph: unknown): graph is ConceptGraph {
  if (!graph || typeof graph !== "object") return false;
  const g = graph as any;
  return Array.isArray(g.concepts) && Array.isArray(g.relationships);
}
```

Called before `mergeGraph` to guard against malformed API responses.

**Affected files:**

- `src/lib/conceptGraph.ts` â€” add version, improve normalize, add dirty tracking, add validateGraph
- `src/types.ts` â€” update `ConceptGraph` interface to include `version`
- `src/lib/graphWriter.ts` (from Friction Point 3) â€” call `validateGraph` before merging

**Verification criteria:**

- "Smith's Practice" and "Smiths Practice" produce the same normalized key

## Friction Point 6: Pure Detection Functions with Hardcoded Logic

### What it is

`src/hooks/useRefineAnalysis.ts` (1,135 lines) contains the audit/suggestion engine. It has pure detection functions that look independently testable but are coupled through hardcoded constants, scoring weights, and regex patterns that can't be configured without editing the file.

**Priority weights are hardcoded** (lines 13â€“70):

```typescript

  // ...

```

These are exported so they're reachable, but there's no mechanism for a brain configuration to say "for this brain, STALE_REMINDER has priority 3 because we don't use reminders heavily".

**Scoring in chatContext.ts uses naive de-pluralization** (line 29):

```typescript

  .toLowerCase()
  .split(/\s+/)
  .map((w) => w.replace(/s$/, "")); // "entries" â†’ "entrie", not "entry"
```

This is both wrong (it produces `"entrie"` not `"entry"`) and unconfigurable. If a brain works in a language where this doesn't apply, there's no override.

**Score weights in chatContext.ts are hardcoded** (lines 33â€“40):


There's no way for different views (Ask vs Feed) to use different scoring profiles, even though they have different priorities (Ask prioritises recent and connected entries, Feed prioritises type and tag matches).

**`SKIP_META` constant is duplicated:**

Two separate sets for the same concept, defined in different files, that will drift over time.

**`detectOrphans` has implicit dependency on links having a specific shape** (useRefineAnalysis.ts line 79):
