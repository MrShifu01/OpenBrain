# Codex Adversarial Codebase Review

Date: 2026-04-30

## Scope

Full adversarial review of the EverionMind codebase, including the Vite/React frontend, Vercel API routes, Supabase migrations, Capacitor shells, test suite, build output, and launch-adjacent project structure.

No code changes were made during the review.

## Verification Run

- `npm run typecheck` passed
- `npm test` passed: 72 files, 466 tests
- `npm run build` passed with CSS/chunk warnings
- `npm run lint` passed with 72 warnings out of a 73-warning budget
- `npm run test:e2e` failed: 9 of 17 tests
- `npm run knip` failed: 4 unused files
- `npm audit --omit=dev` found 0 vulnerabilities

## Executive Summary

The codebase has a solid foundation: TypeScript passes, unit tests are broad, production builds, npm audit is clean, RLS exists across most Supabase tables, auth middleware is centralized, API responses use security headers, and Gmail token encryption shows the right pattern already exists.

The release risk is still meaningful. The highest-risk issues are a real search/runtime crash, OAuth state and token handling, plaintext Calendar OAuth token storage, failing E2E coverage, duplicate Supabase migration numbers, oversized production chunks, CSS optimizer warnings, and a warning budget that is nearly exhausted.

## Detailed Findings

### P0: OmniSearch can crash the app

Location: `src/components/ui/command.tsx:43`

`CommandDialog` renders `CommandInput` and `CommandList` children without wrapping them in the `cmdk` `Command` root. E2E confirms a runtime crash:

```text
TypeError: Cannot read properties of undefined (reading 'subscribe')
```

Impact: search shortcuts and mobile search can break the app shell.

Recommendation: wrap command dialog children in the `Command` component/root expected by `cmdk`, then rerun the Playwright search specs.

### P1: OAuth state is forgeable and auth tokens are passed in URLs

Locations:

- `api/gmail.ts:99`
- `api/calendar.ts:47`
- `src/components/settings/CalendarSyncTab.tsx:61`
- `src/components/settings/GmailSyncTab.tsx:132`

Gmail and Calendar OAuth state is plain base64 JSON rather than signed or nonce-bound. The client also passes Supabase access tokens in query strings while starting OAuth.

Impact: callback CSRF/account-linking risk, plus bearer tokens can leak through browser history, logs, proxies, analytics, and referrers.

Recommendation: start OAuth through an authenticated POST/GET that does not place bearer tokens in the URL. Generate a one-time state nonce, store it server-side or HMAC-sign it with expiry, and verify it on callback before writing integrations.

### P1: Calendar OAuth tokens are stored plaintext

Locations:

- `api/calendar.ts:98`
- `api/calendar.ts:164`
- `api/calendar.ts:203`

Gmail tokens use app-layer encryption via `api/_lib/gmailTokenCrypto.ts`, but Calendar writes and refreshes raw provider tokens.

Impact: DB/service compromise exposes long-lived Calendar refresh tokens.

Recommendation: generalize `gmailTokenCrypto` into provider-token crypto, encrypt Calendar access and refresh tokens, and migrate existing Calendar rows.

### P1: E2E suite is failing

`npm run test:e2e` failed 9 of 17 tests.

Observed failure classes:

- Repeated Radix accessibility errors about missing `DialogTitle`
- Search runtime crash from `cmdk`
- Scenario tests failing around capture/delete/onboarding/schedule/search flows
- Multiple `429` warnings during E2E around timezone preference sync

Impact: release confidence is materially lower than the unit suite suggests.

Recommendation: fix the `cmdk` crash first, then dialog accessibility warnings, then stabilize scenario tests and local rate-limit handling.

### P1: Duplicate Supabase migration versions

Locations:

- `supabase/migrations/004_push_notifications.sql`
- `supabase/migrations/004_user_memory.sql`
- `supabase/migrations/058_canonical_schedule_fields.sql`
- `supabase/migrations/058_user_profiles_onboarded_at.sql`

Impact: depending on migration tooling, duplicate numeric versions can block or mis-order deploys.

Recommendation: rename into unique monotonic versions before relying on clean environment replay.

### P2: Bundle size is too heavy

Build output includes large chunks:

- `exceljs.min`: about 930 kB
- `pdf.worker`: about 2.18 MB
- `pdf`: about 405 kB
- several app chunks over 300 kB

Impact: slower cold load, especially on mobile, for a mobile-first capture product.

Recommendation: lazy-load import/export parsers and PDF tooling only when those panels are opened.

### P2: CSS import ordering is broken

Build reports six CSS optimizer warnings because `@import` appears after other rules in design-family CSS files.

Impact: fonts may not load consistently after optimization.

Recommendation: move all CSS `@import` rules before any style rules, or load fonts centrally.

### P2: React hook warnings are close to becoming hidden bugs

Locations include:

- `src/Everion.tsx:258`
- `src/views/ChatView.tsx:38`
- `src/hooks/useCaptureSheetParse.ts:530`

The lint budget is at 72 warnings out of 73, including missing dependencies, set-state-in-effect warnings, explicit `any`, and skipped React Compiler optimizations.

Impact: stale closures and rendering behavior bugs can hide behind a passing lint command.

Recommendation: reduce warnings below 20, then lower the warning budget until CI can fail on new warnings.

### P2: Private user content is widely cached in plaintext browser storage

Locations include:

- `src/lib/entriesCache.ts:28`
- `src/hooks/useChat.ts:42`
- `src/lib/crypto.ts:144`

Entries, chat history, concept graphs, learning summaries, offline ops, and session vault keys touch `localStorage` or `sessionStorage`.

Impact: XSS or malicious extensions get a large privacy payoff.

Recommendation: add opt-in caching, TTLs, encryption for sensitive caches, and a private-mode setting.

## Positive Findings

- TypeScript passes cleanly.
- Unit coverage is broad and fast: 466 passing tests.
- Production build completes.
- `npm audit --omit=dev` reports no known production dependency vulnerabilities.
- Most data access goes through centralized API/auth helpers.
- Supabase RLS exists across most user-owned tables.
- Gmail token encryption already establishes a usable pattern for Calendar token hardening.
- Security headers and CSP are configured in `vercel.json`.

## Recommended Fix Order

1. Fix OmniSearch `cmdk` composition and rerun Playwright.
2. Replace OAuth URL-token/state handling.
3. Encrypt Calendar tokens and migrate existing rows.
4. Rename duplicate Supabase migrations and replay from a clean DB.
5. Burn down E2E failures before touching polish.
6. Split heavy import/PDF chunks.
7. Reduce ESLint warnings below 20, then make warnings fail CI.
