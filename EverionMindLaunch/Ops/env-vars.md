# Environment variable inventory

Single source of truth for every `process.env.*` and `import.meta.env.*` key the app reads. Generated by grep over `api/` + `src/` on 2026-05-04. Re-run `grep -rEho "process\.env\.[A-Z_][A-Z0-9_]+|import\.meta\.env\.[A-Z_][A-Z0-9_]+" api/ src/ | sort -u` to refresh.

## Where each goes

| Surface | Source |
|---|---|
| **Vercel project env** | All server-side keys (`api/*.ts`). Set via `vercel env add` or dashboard. |
| **Vite build env** (compiled into client) | `VITE_*` keys. `.env` locally; Vercel project env in prod. **Public — assume baked into the JS bundle.** |
| **GitHub Actions secrets** | Cron + CI workflows (Resend, Supabase service role for backups, Sentry releases, weekly-roll-up). |
| **Local `.env`** | Mirror of the above for `npm run dev`. Never committed. |

## Server-side (api/*)

### Supabase
- `SUPABASE_URL` — project base URL (`https://wfvoqpdfzkqnenzjxhui.supabase.co`)
- `SUPABASE_SERVICE_ROLE_KEY` — bypasses RLS. Never ship to client. Used by every Vercel function.

### AI providers
- `GEMINI_API_KEY` — primary LLM + embedding provider (project default)
- `GEMINI_MODEL` — chat default (e.g. `gemini-2.5-flash`)
- `GEMINI_PRO_MODEL`, `GEMINI_STARTER_MODEL`, `GEMINI_PRO_CHAT_MODEL`, `GEMINI_STARTER_CHAT_MODEL` — tier-specific overrides
- `GEMINI_PERSONA_EXTRACTOR_MODEL`, `GEMINI_AUDIT_RULES_MODEL`, `GEMINI_REJECTED_DISTILLER_MODEL`, `GEMINI_GMAIL_DISTILLER_MODEL` — per-task overrides
- `ANTHROPIC_API_KEY` — Pro/Max managed tier (currently NOT configured, see CLAUDE.md)
- `ANTHROPIC_MODEL` — `claude-haiku-4-5-20251001` default
- `GROQ_API_KEY` — fallback / experimental

### Auth + OAuth
- `OAUTH_STATE_SECRET` — signs OAuth state cookies
- `OAUTH_TOKEN_ENCRYPTION_KEY` — encrypts third-party access/refresh tokens at rest
- `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `GOOGLE_REDIRECT_URI` — Google OAuth (sign-in + Gmail)
- `GMAIL_REDIRECT_URI` — Gmail-scope-specific redirect (separate from sign-in)
- `GMAIL_TOKEN_ENCRYPTION_KEY` — encrypts Gmail refresh tokens at rest
- `MICROSOFT_CLIENT_ID`, `MICROSOFT_CLIENT_SECRET`, `MICROSOFT_REDIRECT_URI`, `MICROSOFT_TENANT_ID` — Microsoft OAuth (Outlook calendar)

### Billing
- `LEMONSQUEEZY_API_KEY`, `LEMONSQUEEZY_STORE_ID` — web subscriptions
- `LEMONSQUEEZY_STARTER_VARIANT_ID`, `LEMONSQUEEZY_PRO_VARIANT_ID`, `LEMONSQUEEZY_MAX_VARIANT_ID` — tier → variant mapping
- `LEMONSQUEEZY_WEBHOOK_SECRET` — verifies inbound webhook payloads
- `REVENUECAT_SECRET_API_KEY` — mobile IAP receipt validation
- `REVENUECAT_STARTER_PRODUCT_ID`, `REVENUECAT_PRO_PRODUCT_ID`, `REVENUECAT_MAX_PRODUCT_ID` — tier → product mapping

### Email + Push
- `RESEND_API_KEY` — outbound email (welcome, invite, daily digest, weekly roll-up)
- `RESEND_FROM` — `from` address (e.g. `noreply@everion.smashburgerbar.co.za`)
- `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT` — Web Push (notification bell)

### Observability
- `SENTRY_AUTH_TOKEN`, `SENTRY_ORG`, `SENTRY_PROJECT` — release tagging only (DSN is on the client)

### Ops / cron
- `CRON_SECRET` — bearer token cron handlers verify before doing work
- `GH_DISPATCH_TOKEN`, `GH_OWNER`, `GH_REPO` — `repository_dispatch` POSTs from the app to trigger workflows
- `ADMIN_EMAIL` — destination for daily/weekly admin digest

### Caching / rate-limit
- `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN` — rate-limiter + idempotency reservation store

### Search tuning
- `SEARCH_THRESHOLD` — vector cosine cutoff for `match_entries_for_user`

### Build flags (Vercel-injected)
- `VERCEL` — `1` when running on Vercel (used by `node-vs-vercel` branches)
- `APP_URL` — public origin (e.g. `https://everion.smashburgerbar.co.za`)

## Rotation cadence

| Key | Rotation rule |
|---|---|
| `SUPABASE_SERVICE_ROLE_KEY` | Rotate via Supabase dashboard if a deploy log or commit was leaked. Update Vercel env + GH Actions secret in same operation. |
| `GEMINI_API_KEY` / `ANTHROPIC_API_KEY` / `GROQ_API_KEY` | Rotate quarterly or on any leak. |
| `LEMONSQUEEZY_WEBHOOK_SECRET` / `REVENUECAT_WEBHOOK_AUTH` | Rotate after onboarding new staff with vendor-dashboard access. |
| `OAUTH_STATE_SECRET` / `OAUTH_TOKEN_ENCRYPTION_KEY` / `GMAIL_TOKEN_ENCRYPTION_KEY` | **NEVER rotate without a re-auth flow** — rotating breaks every stored OAuth/Gmail token. If forced (incident), bump version (`*_v2`) and have client UI re-auth users on next sign-in. |
| `CRON_SECRET` | Rotate on any leak. Coordinate with workflow secret update. |
| `VAPID_*` | Don't rotate without invalidating every existing push subscription (browser-side). |
| `VITE_*` | Treated as public from day one. No "rotation" — just don't put real secrets here. |

## Adding a new env var — checklist

1. Decide: server-only or client-bundled? Client-bundled MUST be `VITE_*`.
2. Add to Vercel project (`vercel env add` or dashboard) for `production`, `preview`, `development`.
3. Add to local `.env` (mirror).
4. If a workflow needs it, add to GitHub Actions repo secrets.
5. Document the key in this file with: purpose, surface (server/client), rotation rule.
6. If sensitive, add to `Ops/incident-response.md` rotation playbook.
